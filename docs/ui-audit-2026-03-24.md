# Scratch UI Audit

Date: 2026-03-24

Canonical reference: [ui-design-language.md](/Users/wayne/Source/scratch/docs/ui-design-language.md)

## Scope and Method

This audit is internal-first and preserves Scratch's existing product DNA: calm, editor-first, dense without clutter, and native-feeling rather than ornamental.

Evidence for this pass came from:

- Source review across the shared UI primitives, editor surfaces, navigation shell, settings, overlays, and feedback components
- State-path review for light/dark mode, pane modes `1/2/3`, focus mode, source mode, preview mode, and transient UI
- A clean frontend production build via `npm run build`

What this pass did not include:

- Full manual GUI walkthrough on macOS across every interaction state
- Multi-platform visual QA on Windows/Linux
- Screenshot regression testing

That means the findings below are implementation-grounded and high-confidence, but a short manual verification pass should still happen before turning the backlog into code.

## Scratch UI Canon

### Visual Thesis

Scratch should feel like a quiet writing workspace:

- the editor is the primary plane
- navigation recedes but remains efficient
- overlays feel deliberate and contained
- controls are dense, tactile, and quiet
- polish comes from rhythm, not decorative chrome

### Surface Hierarchy

Use this hierarchy consistently:

1. Editor canvas
2. Supporting panes
3. Utility chrome and inline tools
4. Menus, popovers, dialogs, and toasts
5. Destructive and error states

Permanent layout should stay flat and restrained. Elevated surfaces should only appear when the UI is transient, interruptive, or selection-driven.

### Geometry

Reduce the shape system to three radii:

- `6px` for inline controls, pills, list items, and compact buttons
- `8px` for standard buttons, inputs, menus, and small popovers
- `12px` for dialogs, command surfaces, and large floating panels

Avoid ad hoc values like `10px` and `18px` unless a surface has a clear, singular reason to break the system.

Use one border model:

- `1px` quiet border for all controls and transient surfaces
- solid separators by default
- dashed separators only for secondary subsections or warning-state internals, not as a default layout motif

Use one shadow ladder:

- no shadow on permanent layout regions
- one menu/popover shadow
- one dialog/palette shadow

### Layout Rhythm

Adopt a consistent 4px spacing grid with stable row heights:

- drag region: `44px`
- pane header content row: `36px` to `40px`
- compact icon button: `28px`
- standard text control: `32px`
- primary CTA: `40px`

Headers across folders, notes, editor, and settings should share the same horizontal rhythm:

- left inset: `16px`
- right inset: `12px`
- title/action gap: `6px`
- action cluster gap: `2px`

### Typography

Use typography to preserve calm hierarchy:

- app and pane titles: `16px`, medium
- settings section titles: `20px`, medium
- primary body copy: `14px`
- supporting copy: `12px` to `13px`
- utility counters, pills, and keycaps: `11px` to `12px`

Large `17px` inputs are acceptable only for command-style surfaces such as the command palette or AI prompt, not for routine settings or small inline tools.

### Icons

Standardize icon treatment:

- standard action icon size: `17px` to `19px`
- compact inline icon size: `14px` to `16px`
- default stroke weight: `1.5` to `1.6`
- use heavier strokes only for very small glyphs

Icons should support scanning, not decorate space. If a label already carries the meaning, the icon should either sharpen affordance or be removed.

### Interaction States

Every interactive control should use the same state language:

- hover: muted fill or stronger text, never both arbitrarily
- focus-visible: one shared ring or outline treatment
- active/selected: muted fill plus subtle emphasis, not a completely separate style family
- disabled: opacity reduction plus cursor and interaction lock
- destructive: tinted surface plus destructive label/icon color

Accessibility rule:

- every actionable control must be reachable by `Tab` unless it participates in a documented roving-tabindex model
- list/grid navigation patterns may use roving tabindex, but isolated buttons and icon buttons should not be removed from the tab order

### Overlays

All transient surfaces should derive from one of three shells:

- menu shell: menu, context menu, sort menu, export menu
- popover shell: search toolbar, link editor, block math editor, suggestion list
- dialog shell: command palette, AI modal, folder icon picker, destructive confirmation

Every dialog shell should share:

- backdrop
- centered container
- title/input/body/footer structure
- `Escape` close behavior
- outside-click behavior
- focus trap
- consistent motion and elevation

### Motion

Use a single motion family:

- fast entrance: `160ms` to `200ms`
- pane/layout transitions: `220ms` to `240ms`
- opacity-only utility transitions: `150ms` to `180ms`

Avoid smooth scrolling for keyboard-driven list selection. It makes dense navigation feel floaty and slower than the underlying interaction.

Avoid mixing unrelated motion curves and durations inside the same workflow.

### Empty, Loading, Error, and Success States

Use the same state grammar everywhere:

- empty: one visual, one headline, one supporting sentence, one action
- loading: inline when possible, blocking only for app initialization or destructive waits
- error: tinted container with explicit next step
- success: quiet confirmation, not celebratory UI

Status semantics should be consistent across editor save state, Git state, AI state, settings errors, and toasts.

## Prioritized Polish Backlog

Severity guide:

- `P1`: system-level inconsistency or user-facing interaction debt
- `P2`: clear polish issue that affects trust, flow, or perceived quality
- `P3`: worthwhile finish work after the system is stable

### P1

#### P1. Normalize interactive primitives and restore keyboard reachability

- Surfaces: shared buttons, icon buttons, toolbar buttons, pane actions, settings back button, editor actions
- Why it matters: Scratch positions itself as keyboard-first, but several core controls are removed from the tab order and the focus model is inconsistent across `Button`, `IconButton`, `Input`, and `Select`.
- Evidence:
  - `src/components/ui/index.tsx` removes `ToolbarButton` and `IconButton` from the tab order with `tabIndex={-1}`
  - `src/components/settings/SettingsPage.tsx` uses `IconButton` for the back button, so that navigation control inherits the same issue
  - `src/components/ui/Button.tsx`, `src/components/ui/Input.tsx`, and `src/components/ui/Select.tsx` each use different focus treatments
- Fix direction:
  - define one focus-visible system
  - keep roving tabindex only where there is a true composite widget
  - return standalone buttons to normal tab order
  - move raw buttons and raw inputs onto shared primitives or explicitly document why they are exceptions
- Solve via: shared primitives
- Phase: structural cleanup

#### P1. Collapse transient UI onto one dialog / popover / menu system

- Surfaces: command palette, AI modal, search toolbar, link editor, block math editor, suggestion list, export menu, sort menus, context menus
- Why it matters: Scratch has multiple transient surface families with different geometry, shadows, focus behavior, and dismissal rules. That makes the app feel less finished than the permanent shell.
- Evidence:
  - `src/components/command-palette/CommandPalette.tsx` implements its own centered floating shell and raw input
  - `src/components/ai/AiEditModal.tsx` does the same with a second modal variant
  - `src/components/folders/FolderIconPickerModal.tsx` is a more structured dialog and should be treated as the stronger reference
  - local menu styles are duplicated in `NoteList`, `FolderTreeView`, `SortMenuButton`, and the editor export menu
- Fix direction:
  - define one menu shell, one popover shell, and one dialog shell
  - centralize surface classes, motion, padding, border, and focus handling
  - migrate command palette and AI modal to the dialog shell
  - migrate local menus and popovers to the shared shells
- Solve via: shared primitives
- Phase: structural cleanup

#### P1. Reduce radius and shadow drift to a small token set

- Surfaces: settings cards, command palette, AI modal, folder icon picker, menus, search toolbar, link editor, toasts
- Why it matters: the app currently mixes `rounded-md`, `rounded-lg`, `rounded-xl`, `rounded-[10px]`, and `rounded-[18px]`, plus several shadow strengths. That weakens the sense of a single product language.
- Evidence:
  - command palette and AI modal use `rounded-xl` and `shadow-2xl`
  - settings rely heavily on `rounded-[10px]`
  - folder icon picker uses `rounded-[18px]`
  - menus and popovers vary between `rounded-md` and `rounded-lg`
- Fix direction:
  - choose the canonical `6 / 8 / 12` radius ladder
  - choose one menu shadow and one dialog shadow
  - update all transient surfaces and settings cards to the reduced set
- Solve via: shared tokens plus local cleanup
- Phase: structural cleanup

#### P1. Reconcile settings with the workspace shell

- Surfaces: settings sidebar, settings sections, segmented controls, cards, section dividers
- Why it matters: the settings area feels like a parallel visual language. The workspace shell is flatter and calmer; settings are more carded, more segmented, and more dashed-divider-heavy.
- Evidence:
  - `src/components/settings/SettingsPage.tsx` matches the shell layout but not the shell's restraint
  - `src/components/settings/GeneralSettingsSection.tsx` and `src/components/settings/EditorSettingsSection.tsx` use rounded cards, segmented controls, and frequent dashed separators that are rare in the core app
- Fix direction:
  - keep settings readable, but pull them closer to the shell's flatter hierarchy
  - reduce card count
  - use section spacing before boxed grouping
  - reserve segmented controls for true mode switches
- Solve via: local surface cleanup guided by shared tokens
- Phase: structural cleanup

### P2

#### P2. Normalize pane headers and utility rails

- Surfaces: folders pane, notes pane, editor titlebar, settings sidebar header, preview window
- Why it matters: the app's basic layout is good, but header padding, icon sizing, badge style, and action density drift enough to register as hand-tuned rather than systematized.
- Fix direction:
  - define one pane-header spec
  - normalize title spacing, badge treatment, icon button size, and action cluster rhythm
  - align settings and editor rails with that spec
- Solve via: shared layout spec plus local cleanup
- Phase: quick wins first, then cleanup

#### P2. Unify in-editor tool surfaces

- Surfaces: link editor, block math editor, search toolbar, suggestion lists, code block toolbar
- Why it matters: these components are visually adjacent to the editor and should feel like one family. Right now they differ in corner radius, padding, button size, and control fidelity.
- Fix direction:
  - use one compact popover shell
  - normalize compact icon button sizing
  - replace bespoke raw controls where possible
  - bring code block language selection up to the same quality bar
- Solve via: shared popover shell
- Phase: structural cleanup

#### P2. Consolidate feedback and status semantics

- Surfaces: save indicator, external-change refresh, Git footer, settings Git errors, AI warnings, toasts
- Why it matters: success, warning, and destructive states are handled differently depending on surface, which reduces trust and makes the app feel less intentional.
- Fix direction:
  - define quiet success, warning, and destructive patterns
  - use one badge/tint approach
  - standardize destructive buttons to use tinted surfaces rather than text color alone
- Solve via: shared state tokens and local cleanup
- Phase: quick wins first

#### P2. Standardize motion timing and reduce floaty keyboard scrolling

- Surfaces: pane transitions, focus/source transitions, search popovers, command palette selection, note selection
- Why it matters: Scratch's tone is calm, but some interactions now feel slightly over-animated or use different timing families.
- Evidence:
  - pane transitions use `240ms`
  - some editor chrome transitions use `400ms`
  - command palette and note selection use smooth scrolling for keyboard navigation
- Fix direction:
  - align transient motion to one family
  - keep pane transitions around `220ms` to `240ms`
  - reduce or remove smooth scrolling during keyboard stepping
- Solve via: shared motion constants and local cleanup
- Phase: quick wins

#### P2. Promote source mode to a first-class editor state

- Surfaces: source mode textarea, source toggle, focus/source transitions
- Why it matters: source mode is useful and aligned with Scratch's audience, but it feels visually bolted onto the richer editor surface.
- Fix direction:
  - give source mode a minimal but intentional shell
  - align padding, focus state, and utility affordances with the editor
  - keep the surface quiet, not more chrome-heavy
- Solve via: local surface cleanup
- Phase: local cleanup

### P3

#### P3. Tighten first-run and empty-state artwork/copy

- Surfaces: folder picker, editor empty state, no-results states
- Why it matters: these surfaces are already good. A small copy and illustration pass would make them feel even more deliberate without changing the product's tone.
- Fix direction:
  - review illustration contrast and icon sharpness across light/dark
  - align empty-state copy cadence across screens
- Solve via: local cleanup
- Phase: finish work

#### P3. Review interface zoom at `90%` and `125%`

- Surfaces: entire app
- Why it matters: the zoom feature is valuable, but because it is implemented with root `zoom`, it deserves a dedicated polish pass for crispness, clipping, and spacing at non-default scales.
- Evidence:
  - `src/context/ThemeContext.tsx` applies zoom to the root element
- Fix direction:
  - manually verify all dense surfaces at `90%`, `100%`, and `125%`
  - correct clipping, text softness, and spacing regressions after the primitive cleanup lands
- Solve via: local QA and follow-up cleanup
- Phase: finish work

## Quick Wins vs Structural Cleanup

### Quick Wins

- normalize focus treatment across `Button`, `Input`, and `Select`
- restore tab access for standalone icon buttons
- unify menu item class names and destructive item treatment
- align pane header spacing and badge treatment
- reduce motion drift and remove smooth keyboard scrolling where it feels floaty
- normalize status tint usage for warning and destructive states

### Structural Cleanup

- build a shared dialog shell for command surfaces and modal flows
- build a shared popover shell for editor-adjacent tools
- build a shared menu shell for sort, export, and context menus
- reduce radii and shadow tokens across the app
- reconcile settings with the core workspace visual system
- migrate bespoke raw controls back onto the shared primitives or formalize them as exceptions

## Strong Patterns Worth Preserving

- The editor-first shell is correct and already has the right restraint.
- Theme and appearance tokens are a strong foundation for a full polish pass.
- The folder icon picker is the best reference for future transient UI cleanup.
- Empty states are generally quiet and avoid SaaS-style clutter.
- Preview mode correctly reuses the editor instead of introducing a parallel UI.

## Recommended Execution Order

1. Shared primitives: focus, tab order, radii, shadows, menu/dialog/popover shells
2. Settings-to-shell alignment
3. Editor-adjacent popovers and command surfaces
4. Status and feedback consolidation
5. Final empty-state and zoom polish pass
